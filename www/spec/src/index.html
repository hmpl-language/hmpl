Introduction

hmpl is a small template language for displaying UI from server to client. It is based on customizable requests sent to the server via fetch and processed into ready-made HTML. The language is syntactically object-based and integrated with JSON5 and DOMPurify.

Reduce the size of your javascript files and display the same UI as if it was written in a modern framework and apply Server-Side Rendering, Static Site Generation, Incremental Static Generation (SSR, SSG, ISG) without robot indexing on any sites without Next.js, Remix, Nuxt.js!

hmpl Object

The global hmpl object provides the primary API for defining templates and handling client-server rendering behavior.

hmpl.compile ( templateString, options )

The compile method creates a renderable template function from an HTML-like string containing request objects. These request objects are evaluated, and the result is injected into the DOM.

1. Let _template_ be the result of parsing _templateString_ using hmpl syntax. 2. If _options_ is provided, apply the options to each internal request object: 1. If an option is overridden by the request itself, use the request’s value. 3. Return a function _templateFn_ which, when called, performs: 1. Evaluate embedded request objects. 2. Send each request via Fetch. 3. Receive and sanitize HTML if needed. 4. Replace request placeholders in the DOM with server responses. Options include: - memo: Cache request results. - autoBody: Automatically attach form data. - allowedContentTypes: Restrict valid response MIME types. - disallowedTags: Remove disallowed HTML tags. - sanitize: Use DOMPurify to sanitize responses.

Template Function

The result of calling hmpl.compile is a function that takes request initialization configuration. It supports:

  • A single RequestInit object
  • An array of IdentificationRequestInit objects
  • A function to generate options from context


const templateFn = hmpl.compile({{ "src": "/api/test" }});
const result = templateFn(); // => { response: HTMLElement, status: 200 }

RequestInit

Template functions accept request configuration similar to the Fetch API. The object may include headers, credentials, method, body, etc.

IdentificationRequestInit

Multiple requests can be provided using an array, each with a unique id and associated configuration object.

RequestInit Function

If a function is passed as the argument to the template function, it will be called with the context of the current request. This allows dynamically computing body, headers, and other properties.

Property Watcher (get)

A get function can be provided to track property changes such as response or status for each request.

hmpl.stringify

This method serializes a JavaScript request object into a string that can be used inside template syntax.

Context

The HMPLInstanceContext allows templates to access the request event, including values from event.target, FormData, etc.

Request Object Syntax

The main construct in HMPL syntax is string interpolation using single braces. Unlike frameworks like Cample which use double braces, HMPL uses the format {${request}} to embed request objects directly into templates.

Request objects are JSON5-formatted and parsed using JSON5.parse. You can use hmpl.stringify to generate valid interpolations programmatically. 


  {{
    "src": "/api/test"
  }}

Prior to execution, the template placeholder will be rendered as an HTML comment:
All <script> tags from server responses are removed by default for security.

Request Object Properties

src

Specifies the endpoint to which the request will be sent. This field is required. 


  {{
    "src": "/api/test"
  }}

method

Specifies the HTTP method for the request. Default is GET. 


  {{
    "method": "POST"
  }}

after

Specifies the event after which the request should be sent. Uses the format ${event}:${selector}.

Selectors are resolved within the template string, not globally.

repeat

Determines whether the request should be repeated each time the event fires. Default is true.

indicators

Defines HTML to be shown for specific request states like pending, rejected, or specific HTTP error codes. 


  {{
    "indicators": [
      {
        "trigger": "pending",
        "content": "<p>Loading...</p>"
      },
      {
        "trigger": "error",
        "content": "<p>Something went wrong</p>"
      }
    ]
  }}

autoBody

Enables automatic generation of FormData from SubmitEvent targets.

Only works with <form> tags for now.

memo

Enables memoization to cache responses and avoid re-rendering the same content. Works only when repeat is true.

initId

Refers to a named or numeric ID in a supplied HMPLRequestInit dictionary to use as configuration.

allowedContentTypes

Lists MIME types allowed in the response. Default: ["text/html"]. A wildcard * allows all text-based types.

disallowedTags

Specifies an array of HTML tags to remove from the server response. Default is an empty array.

sanitize

When set to true, the server's HTML response will be sanitized using DOMPurify.
